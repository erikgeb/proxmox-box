# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Infrastructure-as-code for bootstrapping a single home-server Proxmox VE 9 (Debian "trixie") box from a fresh install into a hardened host running a self-hosted application stack (Immich, Syncthing, a Caddy reverse proxy doing Let's Encrypt wildcard TLS, planned VaultWarden) backed by encrypted storage. There is no application source code here — only Ansible playbooks that drive an external Proxmox host over SSH (as root via sudo). IaC here is for documentation and possible future reproducibility; it is not a production-grade reproducible build.

The README's `# How to run` section is the canonical end-to-end runbook and ordered sequence of commands. The live backlog is `BACKLOG.md` at the repo root (prioritized P1–P6 plus nice-to-haves) — treat it as the single source of truth for open work; the README's `# Backlog` section links to it.

## Intent & threat model

- **Single home box, owner-operated.** This serves one household; the owner is physically present often, so availability is not critical — manual steps at boot (unlocking storage) are acceptable, not a defect to automate away.
- **Not exposed to the internet, with ONE deliberate exception: the WireGuard VPN.** Services have no public reverse proxy / HTTP exposure and stay LAN-only by default — do not add public exposure without an explicit decision. The single, explicitly-decided exception is the WireGuard VPN in LXC 102 (see topology + sequencing): it opens one inbound, port-scan-silent UDP port so the owner can reach the LAN services remotely. Because the ISP uses **CGNAT (no public IPv4)**, the tunnel is reached over **IPv6** — the LXC gets a global v6 address (GUA) and the router opens an inbound v6 firewall *pinhole* (no DNAT; v6 is not NATed). A client can only connect when its own network has IPv6 (cellular yes; IPv4-only Wi-Fi no) — the documented fallback for that gap is a relay (Tailscale / a VPS).
- **The VPN trades at-rest encryption for unattended availability (deliberate).** Unlike everything else, the WireGuard LXC autostarts on boot (`onboot=1`) with its keys/config on the **unencrypted** rootfs, so it comes back after a reboot WITHOUT the manual LUKS unlock. That is the whole point: if family power-cycle the box while the owner is away, the VPN returns on its own and is the channel through which the owner runs `unlock_storage.yaml` *remotely*. Cost: a stolen box leaks the WG server key + peer list (low value vs. the still-LUKS-protected photos/vault; the keys regenerate cheaply by re-running `10` + re-enrolling peers). This is why `unlock_storage.yaml` is intentionally NOT modified to start 102.
- **Encryption protects against physical theft, not a live compromise.** The data disk is LUKS2-encrypted so a stolen drive/box leaks nothing at rest. It is unlocked manually after every boot (`unlock_storage.yaml`) — there is intentionally no stored key or auto-unlock, since that would defeat the theft protection. A running, unlocked box offers no at-rest protection; that is an accepted limitation given the LAN-only posture.
- **The Let's Encrypt certs are for in-home HTTPS** (trusted certs on LAN hostnames), not because anything is public. Issuance/renewal is handled by Caddy in LXC 100 via the **ACME DNS-01** challenge (OVH provider) — chosen because there is no inbound HTTP reachability — and Caddy auto-renews. A single wildcard cert (`*.<base-domain>`) covers all services, including the future VaultWarden host.

## Service topology (intended)

- **`docker-srv` LXC (id 100) — shared Docker host** running Immich (+ machine-learning) and Syncthing via `docker compose` (`ansible/resources/docker-compose.yml`). These are media/sync apps that benefit from the iGPU passthrough and share the encrypted volume; co-locating them in one Docker LXC is the convenience tier.
- **VaultWarden LXC (id 101) — isolated, security tier.** Kept in its own container, separate from the media stack, because it holds the password vault — the highest-value secret on the box. The goal is blast-radius isolation: a compromise of the larger media stack should not reach the vault. Treat 100 and 101 as different trust zones. **Decision: run VaultWarden as a native binary in the LXC, NOT via Docker** — this avoids enabling `nesting` (which loosens LXC confinement) on the one container that most needs to stay tight.
- **WireGuard VPN LXC (id 102) — remote-access tier, third trust zone.** A self-hosted WireGuard subnet router for `192.168.0.0/24`: a remote client (10.13.13.0/24) reaches Immich/Syncthing (100), VaultWarden (101) and the Proxmox UI as if on the LAN. Unprivileged, **no nesting** (the kernel WireGuard module runs from the host — no `/dev/net/tun`); static v4 `192.168.0.55` + `ip6=auto` GUA; **`onboot=1` and config on the unencrypted rootfs** (`/etc/wireguard`, NOT a bind mount of the encrypted volume) so it autostarts unattended — see the threat-model trade-off above. Return-path routing uses **MASQUERADE** on the LXC's `eth0`, so LAN hosts (whose gateway is the router, not this box) can reply to VPN clients — at the cost that services see all VPN traffic as coming from `.55` (lost per-client source IP; affects Caddy logs / per-client ACLs). A connected client has full-LAN reach, **including the vault zone** — accept this blast radius or scope client `AllowedIPs` per device.
- **TLS / reverse proxy: Caddy in the Docker LXC (100)** terminates HTTPS for the LAN and routes by hostname to the local media apps (and, once it exists, across to VaultWarden on 101). It is a service in the same compose project (`ansible/resources/docker-compose.yml`, built from `ansible/resources/caddy/Dockerfile` because the stock image lacks DNS plugins), configured by `ansible/resources/caddy/Caddyfile.j2` (a Jinja2 template — `05` renders it at deploy time, substituting the `@vault` route's cross-zone IP from `vaultwarden_ip`; Caddy's `{$VAR}` placeholders pass through and resolve at runtime), with its OVH DNS-01 credentials + base domain in `ansible/resources/caddy.env` (gitignored; template `caddy.env.example`; `05` deploys it to the **encrypted volume** at `secure-storage/secrets/caddy.env` — NOT the container rootfs — because the OVH credential has value beyond the box). Cert/account keys live on the encrypted volume (`/mnt/storage/caddy_data`). Proxying 101's traffic through a proxy in the weaker trust zone is acceptable for LAN-only, but keep the VaultWarden admin path in mind when hardening. **LAN DNS must point the hostnames at LXC 100's IP** — that part is manual (router/Pi-hole/hosts), not automated here.

`ansible/resources/docker-compose.yml` runs the full Immich stack — `immich-server`, `immich-machine-learning`, `database` (vector-enabled PostgreSQL, data in `/mnt/storage/immich_db`), `redis` (Valkey) — plus Syncthing and the Caddy reverse proxy. The DB password is read from a `.env` file beside the compose file; `05_deploy_docker_stack.yaml` generates it from `.env.example` with a strong random `DB_PASSWORD` directly inside the container on first deploy (it is never regenerated, since Postgres bakes it into its data dir on init). Caddy reads `caddy.env` (user-supplied OVH credentials, copied by the deploy playbook onto the encrypted volume at `/mnt/storage/secrets/caddy.env` — the compose `env_file` references it by that absolute path). The postgres/valkey image tags track upstream and may need bumping against the current official Immich release compose. Every service caps its JSON logs (`10m × 3` via the `*default-logging` anchor) so container logs can't grow unbounded on the LXC rootfs (the boot disk, on `local-lvm` — not the encrypted volume).

**Syncthing storage.** Syncthing has **two** bind mounts: `/mnt/storage/syncthing_data:/config` (its own DB/keys/certs) and `/mnt/storage/syncthing_share:/data` (the user-synced data it replicates). When the Syncthing web UI asks for a sync **Folder Path**, it must be a path under `/data` (`/config` is internal state, not for user data). Canonical paths: **`/data/personal`** for personal files, **`/data/vault_backups`** for the VaultWarden `.age` backups (host path `-e syncthing_drop_dir=/mnt/pve/secure-storage/syncthing_share/vault_backups` to `backup_vaultwarden.yaml`), and **`/data/immich_db_backups`** for the Immich DB `.age` dumps (host path `-e syncthing_drop_dir=/mnt/pve/secure-storage/syncthing_share/immich_db_backups` to `backup_immich.yaml`, and to `update_docker_stack.yaml` when bumping Immich). The service uid/gid is pinned to `911` (`PUID`/`PGID`) so it maps to host `100911`, which is why `04` chowns `syncthing_share` to `100911`. Create each folder in the Syncthing UI **before** writing into it from the host (so Syncthing owns the dir and can write its `.stfolder` marker); the host-root backup playbook then writes into a Syncthing-owned dir without issue.

## Tooling

- **Ansible** is the only tool — it drives all host-level config *and* guest provisioning over SSH. There is no OpenTofu/Terraform: the Docker LXC is created with `pct` run as root, because Proxmox restricts bind mounts and device passthrough to `root@pam` and a privsep API token cannot perform them — running `pct` locally as root sidesteps the API entirely.
- Install Ansible collection deps once: `ansible-galaxy collection install -r requirements.yaml`. Note `requirements.yaml` lists only `community.proxmox`, but playbooks also use `community.general`, `community.crypto`, and `ansible.posix` — these must be available too. (Guest provisioning uses raw `pct`/`pvesm` via `ansible.builtin.command`, not the `community.proxmox` API module, to keep root-only operations working.)
- `ansible.cfg` sets the default inventory to `./inventory.ini` and pins `interpreter_python = /usr/bin/python3`.

## Architecture & sequencing

The setup is a strict ordered pipeline; steps depend on artifacts created by earlier steps:

1. **`01_bootstrap_and_harden.yaml`** — the only playbook that targets a raw root login. It runs over `ansible.builtin.raw` (no Python on the host yet), rewrites APT to the free `pve-no-subscription` repos, installs Python + proxmoxer, creates the `ansible-worker` user with passwordless sudo and your `~/.ssh/id_ed25519_ansible.pub` key, then disables root SSH. **Every subsequent playbook assumes this user exists** (`remote_user: ansible-worker`, `become: yes`) and connects via the `proxmox` inventory group.
2. **`03_setup_encrypted_disks.yaml`** — destructive. Wipes `target_disk`, creates one GPT partition, a LUKS2 (argon2id) container `ssdcrypt_data`, ext4, mounts at `/mnt/pve/secure-storage` with `noauto` (so a missing passphrase never blocks boot), creates `immich_data`/`immich_db`/`syncthing_data`/`syncthing_share`/`vaultwarden_data`/`caddy_data` dirs (`immich_db` holds the Immich PostgreSQL data, `caddy_data` the ACME account + cert keys, `syncthing_data` is Syncthing's own state vs. `syncthing_share` the user-synced data it replicates — see the Syncthing note below), and registers it as Proxmox storage `secure-storage` with `--is_mountpoint 1` (so PVE refuses to write to the unencrypted root disk when the volume is unmounted).
3. **`04_provision_docker_lxc.yaml`** — creates the `docker-srv` LXC (id `100`) via `pct` run as root: unprivileged, nesting enabled, rootfs on `local-lvm` (NOT the encrypted volume), `--onboot 0`, bind-mounts `secure-storage` to `/mnt/storage` (`--mp0`), and passes through `/dev/dri/renderD128` (Intel iGPU) for transcoding (`--dev0`, skipped with a warning if the device is absent). Guards: fails fast if `/mnt/pve/secure-storage` isn't currently mounted (else the bind mount would capture the empty unencrypted dir); `pct create` is gated on a `pct status` existence check; `mp0`/`dev0` are reconciled every run via idempotent `pct set`. It also `chown`s the Docker data dirs (`immich_data`/`immich_db`/`syncthing_data`/`caddy_data`) to `100000:100000` (host uid that this unprivileged LXC's root maps to) so the bind-mounted, otherwise-`nobody` dirs are writable; each image's root-stage entrypoint then sub-chowns to its own uid (e.g. Postgres → host `100999`, syncthing → host `100911`). `syncthing_share` is the exception: it's chowned **directly** to `100911:100911` (the Syncthing service uid `911`, pinned via `PUID`/`PGID` in `docker-compose.yml`), because — unlike `/config` (`syncthing_data`) — the linuxserver image does not auto-chown data mounts, so Syncthing must own `/data` itself to create folder markers and write received files. The claim is gated on the dir still being owned by host root (`uid == 0`), so it runs only pre-first-run and is a no-op on re-runs once a service has taken a dir over — forcing `immich_db` back to `100000` would otherwise break an initialized Postgres ("data directory has wrong ownership"). (`vaultwarden_data` is left to `06`.)
4. **`05_deploy_docker_stack.yaml`** — deploys the app stack into the Docker LXC (id `100`) over `pct push`/`pct exec` (not SSH-into-guest, matching the root-only `pct` convention). Guards that `/mnt/pve/secure-storage` is mounted and the container is running (starts it if not). Installs Docker CE + the compose plugin from Docker's apt repo inside the container, gated on a `command -v docker` probe. Pushes `docker-compose.yml` + `.env.example` to `/opt/docker-stack`; on first run only, generates `/opt/docker-stack/.env` from the example with a strong random `DB_PASSWORD` (never rotated afterward — Postgres bakes it into its data dir on init, so re-running must not change it). Also pushes the Caddy resources (`caddy/Dockerfile`; `caddy/Caddyfile.j2` rendered via `ansible.builtin.template` — the `@vault` IP comes from `vaultwarden_ip` — and pushed as plain `Caddyfile`) — failing fast if `caddy.env` is absent — and ensures `/mnt/storage/caddy_data` exists. The user-supplied `caddy.env` (OVH credentials — valuable beyond the box) is NOT pushed to the rootfs: it is copied to the **encrypted volume** at `secure-storage/secrets/caddy.env` (dir + file owned `100000`, mode `0700`/`0600`; created by `05`, not `03`), the compose `env_file` reads it at the absolute path `/mnt/storage/secrets/caddy.env`. Then `docker compose up -d` (which builds the custom Caddy image on first run).
5. **`06_provision_vaultwarden_lxc.yaml`** — creates the `vaultwarden` LXC (id `101`) via `pct` run as root: unprivileged, **nesting NOT enabled** (kept tight — this is the security tier), no GPU, rootfs on `local-lvm`, `--onboot 0`, and bind-mounts **only** the `vaultwarden_data` subdir of `secure-storage` to `/var/lib/vaultwarden` (`--mp0`) so 101 cannot see the media data. Same mount/existence guards as `04`; also `chown`s the host data dir to `101000:101000` so the in-container service user (uid 1000, default unprivileged id-shift) can write.
6. **`07_deploy_vaultwarden.yaml`** — deploys VaultWarden as a **native binary** (no Docker, no nesting) into LXC `101` over `pct push`/`pct exec`. Extracts the `vaultwarden` binary + matched `web-vault` from the pinned official `vaultwarden/server:<ver>` image using LXC `100`'s Docker engine (deploy-time-only dependency), version-gated by a `VERSION` stamp. Installs the dynamically-linked libs (`libssl3`/`libpq5`/`libmariadb3`/`ca-certificates`) and verifies with `ldd` (fails on `not found`). Creates the `vaultwarden` user (uid/gid 1000), writes `/etc/vaultwarden/vaultwarden.env` (signups off; `DOMAIN` derived from `caddy.env`'s `BASE_DOMAIN`; `ADMIN_TOKEN` generated once as an Argon2id hash, plaintext printed once), and installs a hardened systemd unit (`ansible/resources/vaultwarden/vaultwarden.service`).
7. **`unlock_storage.yaml`** — run after every reboot. Prompts for the LUKS passphrase, reopens + mounts `secure-storage` (`-e target_disk=...`, same NVMe-aware partition derivation as setup), then `pct start`s the dependent containers (Docker `100`, VaultWarden `101`) — each gated on a `pct status` probe, so a not-yet-provisioned (or already-running) container is skipped instead of erroring.
8. **`update_vaultwarden.yaml`** — recurrent (lives in `ansible/` root like `unlock_storage.yaml`, self-contained). Upgrades the native VaultWarden binary: resolves the target version (latest GitHub release, or `-e target_version=`), no-ops if already installed, snapshots the vault data + current binary/web-vault to `secure-storage/vaultwarden_backups/`, re-extracts the target (same path as `07`), `ldd`-guards, restarts, and health-checks `/alive` — **auto-rolling back** binary + web-vault + data via a `block`/`rescue` if the new version fails to come up. The Caddy `@vault` block in `caddy/Caddyfile.j2` routes `vault.<base-domain>` to the VaultWarden LXC (`vaultwarden_ip`, default `192.168.0.54`, port 8000 — cross-zone by IP, rendered at deploy time); applied by re-running `05` + reloading Caddy.
9. **`backup_vaultwarden.yaml`** — recurrent (lives in `ansible/` root, self-contained). The only data backup in the repo, and it covers the vault. The rest is backed up off-box by the owner's personal-machine → 2×USB routine (not automated here): Syncthing files via normal replication, and Immich photo originals via a **one-way, read-only `rsync` pull** of the managed library from the box (Immich stays the source of truth — mobile auto-backup and dedup require the managed library, and a two-way sync would corrupt the DB↔file mapping). The Immich DB (`immich_db`) — curated metadata like albums/tags/named-people PLUS the expensive face/search indexes; all regenerable from originals but at hours of re-indexing cost — has its own encrypted backup (`backup_immich.yaml`, below) and optionally Immich's built-in periodic `pg_dump` via `bootstrap/99_optional_immich_db_backup.yaml` (into `immich_data/backups/`, swept up by the photo rsync). That playbook is the one exception to the "Ansible-over-SSH-as-root against the `proxmox` group" rule: it runs against `localhost` and talks to the Immich HTTP API on the LAN, doing a non-destructive GET→merge-`backup.database`→PUT so the Settings UI stays editable (the `IMMICH_CONFIG_FILE` alternative would lock the whole UI and reset omitted keys). It takes a consistent snapshot (briefly `systemctl stop`s VaultWarden, `cp -a` to a staging dir on the encrypted volume, restarts — wrapped in `block`/`rescue` so the service always comes back), then pipes `tar | age` (no plaintext tarball ever hits disk) to write `vault-<ts>.age` into the **required** `-e syncthing_drop_dir=` (a Syncthing-shared folder, so it auto-syncs and rides the USB backup), then prunes with a **tiered GFS scheme** meant for a **daily cron** (`keep_daily`/`keep_weekly`/`keep_monthly`, defaults 7/4/12): all backups of the last N backup days + the newest per ISO week + the newest per month, computed from the **filename dates** (stateless; a missed day shifts the window instead of emptying a tier; unparsable names are never deleted). Encryption is **age asymmetric**: on first run it generates a keypair, persists only the **public** key at `secure-storage/vaultwarden_backups/age-recipient.pub` (non-interactive encryption thereafter) and **prints the private key once** for offline storage — losing it makes every backup undecryptable, and keeping it off-box is what preserves the theft protection. Restore is automated by `recovery/recover_vaultwarden.yaml` (below).
10. **`recovery/recover_vaultwarden.yaml`** — recovery (lives in `ansible/recovery/`, self-contained). The inverse of `backup_vaultwarden.yaml`: restores the vault from a **previous backup** chosen via the **required** `-e backup_path=` (a path on the **host**, not the controller). It accepts either backup shape this repo makes — an age-**encrypted** `vault-<ts>.age` archive (requires `-e age_identity_file=`, a **controller** path to the OFFLINE private key) or an **unencrypted** pre-upgrade snapshot `data/` dir from `update_vaultwarden.yaml` (no key) — choosing the mode by the `.age` extension. It is **destructive** (overwrites the live `vaultwarden_data`, losing every change since the backup) so it refuses to run without an interactive `RESTORE` confirmation (skip with `-e force=true`). Safety order: decrypt/extract into a scratch staging dir and **validate** it (`db.sqlite3` present) BEFORE touching anything (a bad path/key/archive fails with the live vault untouched, no downtime); then stop the service, snapshot the **current** vault to `vaultwarden_backups/pre-restore-<ts>/data` (so the restore is itself reversible — re-run with that as `backup_path` to undo), swap the backup in (`chown -R 101000:101000`), restart, and health-check `/alive` — a `block`/`rescue` **auto-rolls the pre-restore snapshot back** if the restored vault fails to come up. On success the pre-restore snapshots are pruned to `keep_pre_restore` (default 5) — scoped to `pre-restore-*` so it never touches `update_vaultwarden.yaml`'s `<ts>_<ver>` upgrade snapshots. The age identity is copied to a transient `0600` host file only for the decrypt and **`shred`ded in `always`** (success or fail), so the private key never lingers on the box. **No client↔server reconciliation exists after a restore** (Bitwarden sync is server-authoritative; clients never push their cache back): entries created since the backup survive only in the pre-restore snapshot and in a client app's local cache — the latter lingers only until the first post-restore **server-side edit** bumps the account revision date, at which point the client's next sync replaces its cache wholesale (a restored older vault's revision date doesn't trigger a re-download by itself); the recovery procedures (undo via the snapshot; airplane-mode export from a client) are in the README's restore section.
11. **`backup_immich.yaml`** — recurrent (lives in `ansible/` root, self-contained). The on-demand/scheduled sibling of `update_docker_stack.yaml`'s pre-bump dump: a logical `pg_dumpall --clean --if-exists` of the Immich DB (no service stop — transactionally consistent on a live Postgres), piped through `gzip | age` (no plaintext on disk) into the **required** `-e syncthing_drop_dir=`, named `immich-db-<ts>_<ver>.sql.gz.age` (the version label is the restore-compatibility floor). It encrypts to the **same recipient** (`secure-storage/immich_db_backups/age-recipient.pub`, generate-once/print-once by whichever playbook runs first), so **one offline key decrypts every Immich DB dump** and both playbooks' artifacts are interchangeable inputs to `recover_immich.yaml`. Both prune the shared `immich-db-*.sql.gz.age` pool to `keep_db_backups` (default 3 in both — keep them aligned or the lower one wins).
12. **`recovery/recover_immich.yaml`** — recovery (lives in `ansible/recovery/`, self-contained). Restores the Immich DB from the **required** `-e backup_path=` (host path); **destructive** (live DB overwritten, photo files untouched; interactive `RESTORE` confirmation, `-e force=true` to skip). Accepts three shapes: an encrypted `*.sql.gz.age` (needs `-e age_identity_file=`, controller path, shredded after), a plain `*.sql.gz` (e.g. Immich's built-in backup), or a raw Postgres **data directory** (e.g. its own `pre-restore-<ts>/data`; same postgres image only). Safety order mirrors `recover_vaultwarden.yaml`: stage + validate the dump (pg_dumpall banner check, search_path `sed` fix) BEFORE any downtime — staging lives on `secure-storage` so the container reads it via the `/mnt/storage` bind mount (chowned to `100000` across the id-shift); then stop `immich-server`/`-machine-learning`/`database` (Caddy/Syncthing stay up), snapshot the current `immich_db` to `immich_db_backups/pre-restore-<ts>/data`, **empty the data dir in place** (bind-mount inode convention), re-init a **fresh** Postgres from the same `.env` and `psql`-load the dump (a fresh cluster is what restores across a failed/partial migration or a Postgres image swap), `docker compose up -d`, and health-check `/api/server/ping` from the controller — `block`/`rescue` **auto-rolls the pre-restore snapshot back** on failure. Pre-restore snapshots prune to `keep_pre_restore` (default 2 — they hold the multi-GB vector indexes). Dump restores only migrate **forward**: deployed Immich must be ≥ the dump's version; to also revert a version bump, re-pin the old version via `update_docker_stack.yaml` first, then restore.

**WireGuard VPN bootstrap (`bootstrap/09_provision_wireguard_lxc.yaml` + `10_deploy_wireguard.yaml`) — an independent track, deliberately NOT gated on the encrypted-storage pipeline.** Unlike `04`–`07`, these have **no `secure-storage` mount guard and no bind mount** (config lives on the LXC rootfs), and the container is **`onboot=1`**, so the VPN is up after a bare reboot before any LUKS unlock — that independence is the design goal (remote unlock; see threat model).
  - **`09`** loads + persists the `wireguard` kernel module on the **host** (`community.general.modprobe`, then a `modinfo` assert — a container cannot load host modules, the single most common failure mode) and `pct create`s LXC `102` unprivileged, **no nesting**, `--onboot 1`, `--net0 ...,ip6=auto` (static v4 `.55` + SLAAC GUA). No `/dev/net/tun` is needed for the kernel path.
  - **`10`** installs `wireguard-tools`/`iptables`/`qrencode` (NEVER the `wireguard` metapackage → pulls DKMS), sets `net.ipv4.ip_forward` via a `sysctl.d` drop-in (namespaced; settable inside the CT), **generate-once** server keypair + `wg0.conf` (built with an `echo`-group, not a heredoc — the cmd is a folded YAML scalar; the `PostUp`/`PostDown` add the FORWARD ACCEPT rules + `MASQUERADE -s 10.13.13.0/24 -o eth0`), enables `wg-quick@wg0`, and health-checks `wg show` + that a global IPv6 exists. Both are generate-once so re-runs never clobber enrolled peers.

Maintenance / upgrade playbooks (recurrent or one-time; not part of the ordered bootstrap pipeline):
- **`bootstrap/08_enable_unattended_upgrades.yaml`** — one-time host setup. Installs `unattended-upgrades` scoped (via an `Origins-Pattern` drop-in that replaces the packaged default) to the **Debian-Security** origin only, with `Automatic-Reboot "false"`, and enables the `apt-daily-upgrade.timer`. Keeps security patches flowing automatically; PVE/Ceph upgrades are deliberately excluded (they go through `update_proxmox.yaml`).
- **`update_proxmox.yaml`** — recurrent (in `ansible/` root). `apt` `update_cache` + `upgrade: dist` + autoremove on the host, then `apt-get update && full-upgrade && autoremove` inside each running LXC guest (gated on `pct status`; this also bumps `docker-ce` in `100`). Never reboots — it only reports `/var/run/reboot-required` (a reboot needs a manual LUKS unlock afterwards).
- **`update_docker_stack.yaml`** — recurrent (in `ansible/` root). The upgrade path for LXC `100`'s compose stack: same mount/container guards as `05`, re-pushes `docker-compose.yml` (so repo tag bumps for postgres/valkey/syncthing/caddy take effect), optionally `sed`s `IMMICH_VERSION` in the **live** `.env` when `-e immich_version=` is passed (the pin lives in the generate-once `.env`, so a repo `.env.example` edit would NOT propagate — this is the one Immich-version gotcha), then `docker compose pull && up -d`. VaultWarden is upgraded by `update_vaultwarden.yaml`, not here. **Pre-bump DB guard:** when (and only when) `-e immich_version=` is passed, it **requires `-e syncthing_drop_dir=`** and first takes a logical `pg_dumpall` of the Immich DB (via `docker exec immich_postgres`, no `-t` so the stream isn't CRLF-corrupted), pipes it straight through `gzip | age` (no plaintext dump ever hits disk), and drops the encrypted `immich-db-<ts>_<oldver>.sql.gz.age` into that Syncthing-shared folder — so it auto-syncs to the personal machine and rides the USB backup, the **same off-box pipeline and age generate-once/print-once flow as `backup_vaultwarden.yaml`** (own recipient at `secure-storage/immich_db_backups/age-recipient.pub`; private key printed once, stored OFFLINE). The bump **aborts if the dump fails** (`pipefail`), so an Immich upgrade never proceeds without a backup. It is encrypted because the dump leaves the LUKS volume (Immich account hashes + personal metadata). A logical dump (not a `cp -a` of `immich_db`) is used because a major bump can also swap the Postgres image (the 2.x→3.0 `pgvecto.rs`→VectorChord cutover), and only a logical dump restores across that. Unlike `update_vaultwarden.yaml` it does **not** auto-roll back (a failed major migration also involves the Postgres image + data dir, so restore is a deliberate step: re-pin the old Immich/postgres, then run `recovery/recover_immich.yaml` on the pre-bump dump — the success output spells it out); dumps prune to `keep_db_backups` (default 3) in the drop folder, a pool shared with `backup_immich.yaml`. The Syncthing folder must be **created in the UI first** (so Syncthing owns it). A plain repo-tag pull (no `immich_version`) skips all of this. After every recreate (bump or not), it `docker image prune -af` + `builder prune -f` inside the LXC to drop the superseded image layers and Caddy build cache that each `pull`/`up -d` leaves behind — the main thing that otherwise creeps the boot disk toward full over successive upgrades (named volumes like `immich-model-cache` are untouched).
- **`check_updates.yaml`** — recurrent, **read-only** (in `ansible/` root, self-contained). Reports which LXC `100` apps have a newer upstream version than is deployed; mutates nothing (no pull/restart/file edit), so it needs only the container running (not storage mounted) and never starts it. Reads **deployed** versions live off the box — Immich from the live generate-once `.env`, postgres/valkey/syncthing from the deployed `docker-compose.yml` — and fetches **latest** via the host's outbound network: Immich/caddy from their GitHub `releases/latest`, syncthing's full `vX.Y.Z-lsNNN` tag from the **LinuxServer.io API** (`api.linuxserver.io`), but the **postgres/valkey** targets from Immich's *own* upstream compose at its latest release tag (those images track Immich, so they should bump alongside it — not to "latest postgres"). syncthing compares the **whole tag**, so a `-lsNNN`-only rebuild (same Syncthing version, but a fresh base-OS/s6/security rebuild — worth pulling) is flagged; caddy is a floating `caddy:2` local build, so only a new **major** is flagged. Comparison is plain string inequality (drift, not ordered semver — the human decides). Output is a table + a one-line "what to run" summary pointing at `update_docker_stack.yaml`.
- **`update_proxmox_cert.yaml`** — recurrent (in `ansible/` root, self-contained). Makes the Proxmox web UI serve a trusted cert instead of its self-signed one, by **reusing Caddy's existing wildcard cert** rather than issuing a new one. Guards that `/mnt/pve/secure-storage` is mounted (the cert lives there), `find`s the `wildcard_*.crt` Caddy wrote under `caddy_data/caddy/certificates/` (preferring the Let's Encrypt **production** issuer dir over a staging one if both exist), and installs it + its key for `pveproxy` via `pvenode cert set --force --restart`. Idempotent: it compares the source cert's SHA-256 fingerprint against the one `pveproxy` currently serves (`/etc/pve/local/pveproxy-ssl.pem`) and only re-installs/restarts on a change. If Caddy hasn't issued the cert yet it reports that and exits via `meta: end_play` without touching Proxmox's cert. **A wildcard `*.<base-domain>` does NOT cover the apex `<base-domain>` itself, nor the raw IP** — so the Proxmox UI must be reached at a **subdomain** (e.g. `https://pve.<base-domain>:8006`) whose **LAN DNS points at the Proxmox HOST's IP** (not LXC 100, where the apps live; that DNS is manual, like the other hostnames). Because Caddy auto-renews and the installed copy is a point-in-time snapshot, **re-run periodically** to refresh it (e.g. monthly cron); re-runs are no-ops until the cert actually rotates.

- **`add_wireguard_peer.yaml` / `remove_wireguard_peer.yaml`** — recurrent (in `ansible/` root, self-contained). The ongoing VPN ops tools. **Add** (`-e peer_name=` + `-e wg_endpoint=`, optional `-e peer_ip=`/`-e wg_client_dns=`): generates a client keypair in-CT, auto-allocates the next `10.13.13.x`, appends a `# <name>`-marked `[Peer]` block to `wg0.conf`, **hot-reloads with `wg syncconf` (existing tunnels are not dropped)**, persists the client `.conf` under `/etc/wireguard/clients/` (CT rootfs, re-displayable) and prints it + a `qrencode` terminal QR. Idempotent on the `# <name>` marker (re-displays instead of duplicating). **Remove** (`-e peer_name=`): `sed`-strips the marker..`AllowedIPs` block, deletes the client `.conf`, `wg syncconf` to drop it live; no-op if absent.
- **`update_wireguard_ddns.yaml`** — recurrent (in `ansible/` root, self-contained). Upserts the `vpn.<base-domain>` **AAAA** at the LXC's current GUA via the **OVH DNS API**, reusing the OVH credentials from `caddy.env` (same provider as Caddy's DNS-01 — signs requests with the app-secret/consumer-key SHA-1 scheme; reads creds controller-side like `07`). Runs a **read-only credential-scope preflight** (`GET /auth/currentCredential`) that asserts POST+PUT on `/domain/zone*` and fails with guidance if the key is too narrow (OVH rights can't be widened post-creation). Idempotent (only PUT/POST + zone refresh when the target differs). The owner's prefix is treated as static, so it's run **once**; put it on cron only if the prefix starts rotating.

**Cross-file coupling to keep consistent** (these magic values are duplicated, not shared):
- Container IDs: `container_id` defaults to `100` in `04_provision_docker_lxc.yaml`, `05_deploy_docker_stack.yaml`, `unlock_storage.yaml`, and `update_docker_stack.yaml`; `vaultwarden_container_id` is `101`. `update_proxmox.yaml` lists all three (`100`, `101`, `102`) in `guest_container_ids` so every guest gets OS patches. The **WireGuard LXC is `102`** (`container_id` in `09`/`10`/`add_`/`remove_`/`update_wireguard_ddns.yaml`) — note it is deliberately **not** in `unlock_storage.yaml` (onboot=1, independent of the unlock). Its config lives on the rootfs at `/etc/wireguard` (no encrypted bind mount); IPs: v4 `192.168.0.55`, VPN subnet `10.13.13.0/24` (server `.1`), ListenPort `udp/51820`.
- Storage path `/mnt/pve/secure-storage` (host) ↔ `/mnt/storage` (container bind mount) ↔ the `/mnt/storage/*` volume mounts in `ansible/resources/docker-compose.yml` (`immich_data`, `immich_db`, `syncthing_data`, `syncthing_share`, `caddy_data`). The disk-setup data dir names, the compose volume mounts, and the storage name `secure-storage` must all agree. Additionally `secrets/caddy.env`: `05` writes it at `{{ secure_storage_path }}/secrets/caddy.env` ↔ the compose `caddy` service's absolute `env_file` path `/mnt/storage/secrets/caddy.env` ↔ `update_docker_stack.yaml`'s fail-fast guard on the same host path — keep the three in step (the `secrets` dir is created by `05`, not `03`, so it appears on existing deployments on the next `05` run).
- **Bind-mount ownership under the unprivileged-LXC id map** (default container 0 → host 100000): host-root-owned dirs appear as `nobody` and are unwritable inside the container, so the host dirs must be re-owned to the host uid the relevant container user maps to. `04` chowns the Docker data dirs to `100000:100000` (container root) but `syncthing_share` to `100911:100911` (the Syncthing service uid `911`, pinned via `PUID`/`PGID` in `docker-compose.yml`); `06` chowns `vaultwarden_data` to `101000:101000` (the native service's uid 1000). Keep these in step with the in-container users (Docker images' own service uids; the `vaultwarden` user pinned to 1000 in `07`).
- The LUKS mapper name `ssdcrypt_data` and the `target_disk`-derived partition must match the real disk. Both `03_setup_encrypted_disks.yaml` and `unlock_storage.yaml` derive `target_partition` from `target_disk` (appending `p1` for nvme/mmcblk, else `1`), so pass the same `-e target_disk=` to both.

`ansible/resources/docker-compose.yml` is the app stack for the Docker LXC, deployed by `05_deploy_docker_stack.yaml`.

## Secrets & gitignored files

`inventory.ini`, `group_vars/all.yaml` (per-deployment variable overrides; template `group_vars/all.yaml.example` — see Conventions), `ansible/resources/.env` (DB password, generated on-box), `ansible/resources/caddy.env` (OVH DNS-01 API credentials + base domain), and `ansible/resources/immich.env` (an Immich admin API key, used only by the optional DB-backup playbook; template `immich.env.example`) are gitignored. Copy `inventory.ini` from `inventory.ini.example` (fill in the host) and `caddy.env` from `caddy.env.example` (fill in the OVH app key/secret/consumer key, endpoint, base domain, ACME email); never commit either. The controller copy of `caddy.env` is the source of truth; on the box it lives only on the **encrypted volume** (`secure-storage/secrets/caddy.env`, placed by `05`) — never on an unencrypted rootfs, because the OVH credential grants DNS-zone control independent of the box (see the README's theft checklist). The SSH automation key is expected at `~/.ssh/id_ed25519_ansible` (passphrase-less, generated per the README).

The **age backup keypair** (`backup_vaultwarden.yaml`) is handled like the VaultWarden admin token: the **public** key is persisted on-box at `secure-storage/vaultwarden_backups/age-recipient.pub`, but the **private** key is printed once and stored OFFLINE by the owner — it is never written to disk on the box (and must not be, or off-box theft protection is lost) and is not in git.

## Conventions

- Playbook filenames use **snake_case** (underscores, never dashes) — e.g. `unlock_storage.yaml`, `05_deploy_docker_stack.yaml` — matching Ansible's required snake_case for roles/collections/variables. Keep new playbooks consistent; do not reintroduce hyphens. (Tool-mandated names like `docker-compose.yml`, `Caddyfile`, `Dockerfile` are exempt, as are runtime identifiers like the `secure-storage` storage name and the `ansible-worker` user, which are not source filenames.)
- Playbook task naming and idempotency style: shell/`command` tasks that may legitimately fail on re-run guard with `failed_when` checking for "already exists" in stderr and set `changed_when` explicitly; non-idempotent imperative steps (`pct create`) are gated on an existence probe (`pct status`) with `failed_when: false`. Follow this pattern for new imperative `pct`/`pveum`/`pvesm` calls rather than relying on module idempotency that doesn't exist for these CLIs.
- Everything is Ansible-over-SSH-as-root. Provision Proxmox guests with raw `pct` (not the `community.proxmox` API module), since the operations this repo needs — bind mounts and device passthrough — are root@pam-only and unavailable to a privsep API token.
- **Deployment-specific values are overridable via gitignored `group_vars/all.yaml` (template: `group_vars/all.yaml.example`) — never hardcode them in play `vars:`.** Play-level `vars:` silently SHADOW inventory group_vars in Ansible's precedence order (and `-e` beats both), so a value a deployment must be able to override (`vaultwarden_ip`, `vaultwarden_ip_suffix`, `syncthing_drop_dir`, `target_disk`) is left OUT of play `vars:` and given its repo default at the use site with `| default('...')` (see the health checks in `recovery/recover_vaultwarden.yaml` / `update_vaultwarden.yaml` and the `--net0` line in `06`). Repo-internal values (container IDs, on-box paths like `backups_root`, ports) stay in play `vars:` — deliberately shadowing group_vars, so a stray group_vars entry with the same name cannot redirect e.g. plaintext on-box snapshots into a synced folder.
- **Never `rm -rf`/`mv`-replace a bind-mount source directory while its container runs — replace the CONTENTS in place.** An LXC `mp0` bind mount is pinned to the source directory's **inode**, not its path. Deleting and recreating the host dir (e.g. `/mnt/pve/secure-storage/vaultwarden_data`, or the Docker data dirs bind-mounted into 100) leaves the running container attached to the unlinked inode: the mount point reads as empty and every write fails `ENOENT`, until the container is restarted — and any "rollback" that repeats the pattern stays just as broken (this bit the VaultWarden restore AND its auto-rollback in one run). Established pattern (`recovery/recover_vaultwarden.yaml`, `update_vaultwarden.yaml`): `find <dir> -mindepth 1 -delete` to empty it (keeps the inode, handles dotfiles), then `cp -a <src>/. <dir>/` and re-`chown`.
- **`pct push`/`pct pull` read/write their source on the *host* filesystem, not the controller.** The Ansible controller (the operator's laptop) is a **different machine** than the Proxmox host (the mini PC); playbooks connect over SSH. So you cannot `pct push <vmid> {{ resources_dir }}/<file>` directly — `resources_dir` (`{{ playbook_dir }}/../resources`) is a controller path that doesn't exist on the host. **Established pattern (already followed by `05_deploy_docker_stack.yaml` and `07_deploy_vaultwarden.yaml`):** stage the file onto the host first with `ansible.builtin.copy` (which runs over SSH, controller→host, e.g. to `/tmp`), then `pct push` from *that* host path, and verify it landed with `&& pct exec <vmid> -- test -f <dst>` (push has been observed to exit 0 without copying). Likewise `pct pull` lands files in the host's `/tmp` (see `07`'s extract-from-Docker step). Follow this when adding any new `pct push`/`pct pull` of a controller-side resource.
